# Процес пошуку: 6 етапів, у кожному — своя воронка

**Дата:** 2026-07-13
**Рішення власника:** етапи живуть на **вакансії**; артефакти підготовки —
**структуровані в БД** з експортом PDF; гейт між етапами — **мʼякий**
(попередження, не блокування); канал повідомлень — **email (Resend)**;
лонг-лист — **кандидати + імпорт з Excel**; публічний бріф — **AI-чернетка
з внутрішнього бріфу**; існуючі дані — **авто-розкладка по етапах**.

---

## Модель

Раніше: `vacancies` → плаский список `pipeline_stages` → kanban.
Тепер: `vacancies` → **`search_phases` (6)** → `pipeline_stages` (кожна стадія
належить етапу через `phase_id`) → kanban всередині розкритого етапу.

| Етап | `kind` | Що всередині |
|------|--------|--------------|
| 1. Підготовка | `preparation` | **без кандидатів**: бріф, стратегія пошуку, план проекту, бріф для кандидатів, матриця компетенцій |
| 2. Лонг-лист і скринінг | `longlist` | Ідентифікований → Контакт встановлено → Скринінг → Пройшов скринінг |
| 3. Шорт-лист | `shortlist` | Інтервʼю по компетенціях → Оцінка і звіт → Збір рекомендацій → У шорт-листі |
| 4. Співбесіди з замовником | `client_interviews` | Надіслано клієнту → Інтервʼю заплановано → Інтервʼю проведено → Фідбек клієнта → Фіналіст |
| 5. Кейси і фінал | `final` | Кейс надіслано → Кейс оцінено → Фінальне інтервʼю → Обраний кандидат |
| 6. Офер | `offer` | Офер готується → Узгоджено з клієнтом → Офер надіслано → Прийнято |

Статус етапу: `pending` → `active` → `done` (`started_at`/`completed_at`
проставляє тригер, клієнт шле лише статус).

**План проекту = планові дати етапів** (`planned_start` / `planned_end`).
Окремої сутності плану навмисно немає.

**Відмова — наскрізний статус заявки** (`applications.status` + `rejections`),
а не колонка в кожному етапі.

**Гейт мʼякий:** БД нічого не блокує. UI попереджає, якщо кандидати вже в роботі,
а етап «Підготовка» не закритий. (Жорсткий гейт лишається на requisition:
вакансію не відкрити без approve вакансії й проекту-батька.)

## Що зроблено (спринт 1)

- Міграція `20260713090000_ats_search_phases.sql`: enum-и `search_phase_kind` /
  `search_phase_status`, таблиця `search_phases` (+RLS дзеркально до
  `pipeline_stages`), `pipeline_stages.phase_id`, етапні колонки в
  `pipeline_stage_template_items`, **новий дефолтний шаблон «MetaVision — 6 етапів
  пошуку»** (21 стадія), функція `mp_seed_vacancy_pipeline(vacancy, template)`
  (security invoker, ідемпотентна) і **backfill** існуючих вакансій:
  `sourced/screening → етап 2`, `interview → етап 3`, `assessment → етап 5`,
  `offer/hired/rejected → етап 6`. Кандидати лишаються на своїх стадіях.
- Хук `use-search-phases.ts`: `useSearchPhases`, `useSeedVacancyPipeline`,
  `useSetPhaseStatus`, `useUpdatePhasePlan`.
- UI: вкладка «Воронка» → **«Етапи пошуку»** (`PhaseNav` — степер із
  лічильниками; розкритий етап показує лише свої стадії), `PreparationPanel` —
  чеклист етапу 1 + дедлайни етапів; переміщення кандидата через select зі
  стадіями, **згрупованими по етапах** (можна і всередині етапу, і на інший).
- Створення вакансії тепер сіє етапи + воронку одним RPC (старий плаский засів
  `seedVacancyStagesDirect` прибрано — він створював стадії без `phase_id`).
